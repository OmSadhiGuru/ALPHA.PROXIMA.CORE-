# Institutional Intelligence Translation Framework v1.0

---

## Preamble

Research does not create intelligence.

Research becomes intelligence only when it changes the institution.

The Alpha Proxima Foundation does not exist to accumulate knowledge. It exists to transform knowledge into enduring institutional capability — capability that persists across technology generations, research domains, and organizational evolution.

This Framework defines the permanent methodology through which every Canonical Research Program is translated into practical institutional intelligence. It is the bridge between what the Foundation learns and what the Foundation becomes.

The Framework answers a single question for every completed research program:

> *What should Alpha Proxima become because this knowledge now exists?*

---

## Foundational Principle

> **The Intelligence Lifecycle:** Research → Canonical Knowledge → Institutional Translation → Engineering → Knowledge Graph → Education → Products → Institutional Memory

Each stage is a transformation, not a summary. The output of each stage is qualitatively different from its input. A research finding is not yet engineering guidance. Engineering guidance is not yet a built system. A built system is not yet institutional memory. The Framework ensures that every stage is explicitly executed — not assumed, skipped, or compressed.

A Research Program that completes without activating this Framework has produced a document, not intelligence.

## The Eight Translation Dimensions

Every Canonical Research Program is evaluated against eight translation dimensions. The Translation Decision Matrix (separate document) determines which dimensions each program activates. Not all programs activate all eight — but all eight must be explicitly evaluated and a conscious decision made to activate or defer each.

---

### Dimension 1 — Institutional Implications

**Question:** How should the Foundation evolve as an institution because this knowledge exists?

**Scope:** Constitutional amendments, charter revisions, new office creation, policy updates, governance changes, decision authority shifts, institutional structure evolution.

**Guiding sub-questions:**
- Does this knowledge reveal that the Foundation's current structure is missing a required capability?
- Does this knowledge change the authority relationships between existing bodies?
- Does this knowledge require a new standing policy or standing order?
- Does this knowledge render an existing institutional assumption incorrect?

**Output types:** Constitutional provision, charter amendment, new charter, new policy, standing order, governance directive.

**Example (RP-002 → Institution):** The discovery that memory is reconstructive, not reproductive, and that dormant documents drift from accuracy has direct constitutional implications — the Foundation's document review cycle is not merely a housekeeping practice; it is the institutional equivalent of memory rehearsal. This implies a minimum review cadence policy is constitutionally warranted.

---

### Dimension 2 — Engineering Implications

**Question:** What systems, tools, or automation should be built because this knowledge exists?

**Scope:** Software systems, APIs, automation workflows, data pipelines, monitoring systems, Vault tooling, LUMIAION capabilities, search and retrieval infrastructure.

**Guiding sub-questions:**
- Does this knowledge identify a process that could be automated?
- Does this knowledge reveal that an existing system is architecturally misaligned?
- Does this knowledge suggest a new tool the Foundation needs to build?
- Does this knowledge define the requirements for a system that does not yet exist?

**Output types:** Engineering Proposal (EP-series), Architecture Decision Record (ADR), system specification, technical requirement document.

**Example (RP-002 → Engineering):** The trisynaptic circuit model (DG pattern separation → CA3 pattern completion → CA1 integration) suggests a Vault search architecture: a pattern separation layer that disambiguates similar queries, a pattern completion layer that retrieves full context from partial cues, and an integration layer that surfaces the most relevant results. This is a concrete architectural requirement for LUMIAION's Vault search.

---

### Dimension 3 — Knowledge Graph Implications

**Question:** What new node types, relationship types, or graph structures does this knowledge require?

**Scope:** New concept nodes, new typed relationship edges, new graph clusters, cross-program connections, metadata schema extensions, graph query patterns.

**Guiding sub-questions:**
- What new concepts does this program introduce that are not yet nodes in the knowledge graph?
- What relationships between existing nodes does this program reveal?
- Does this program create a new cluster of related concepts that should be graphed together?
- What cross-program edges does this program enable with prior programs?

**Output types:** Knowledge Graph Update Recommendation (per ISR-001 template), new concept notes, graph schema extension.

**Note:** This dimension is always active. Every Canonical Research Program produces new nodes and edges.

---

### Dimension 4 — Educational Implications

**Question:** How should this knowledge become curriculum, teaching, or learning architecture?

**Scope:** Concept access maps, learning sequences, prerequisite structures, teaching notes, workshop designs, onboarding materials, Founder learning programs.

**Guiding sub-questions:**
- What is the minimum conceptual vocabulary a Foundation member needs to engage with this knowledge?
- In what order should concepts be introduced to a learner new to this domain?
- What are the most common misconceptions this knowledge corrects?
- How should this knowledge be taught to an AI cognitive function being onboarded to this domain?

**Output types:** Learning sequence, concept access map, prerequisite graph, teaching note, workshop design, AI function briefing.

**Note:** The Canonical Glossary produced during research is a prerequisite for educational translation, but it is not itself educational design. Educational design adds sequence, pedagogy, prerequisite structure, and access for non-experts.

---

### Dimension 5 — Product Implications

**Question:** Does this knowledge enable a future application, service, or external capability?

**Scope:** Products and services the Foundation might build for external users, internal tools that become externally valuable, capabilities that constitute a distinctly Alpha Proxima intellectual contribution to the world.

**Guiding sub-questions:**
- Is there a persistent problem in the world that this knowledge addresses better than existing solutions?
- Does the Foundation's unique multi-disciplinary, multi-function research approach produce insights unavailable elsewhere?
- Is there an external organization, community, or individual who would benefit from this knowledge in practical form?
- What would a 5-year product vision built on this research look like?

**Output types:** Product vision document, product specification, service design, IP documentation.

**Note:** Not every program activates this dimension. The Foundation must explicitly evaluate whether a product opportunity exists — not assume one does, and not dismiss the possibility without evaluation.

---

### Dimension 6 — AI Implications

**Question:** How should LUMIAION, ATHENA, SOHMA, or future cognitive functions inherit this knowledge?

**Scope:** Changes to how cognitive functions operate, LUMIAION context briefing design, cognitive function briefings for domain knowledge, parametric knowledge integration strategies, function-specific capability upgrades.

**Guiding sub-questions:**
- Which cognitive functions perform differently if they operate with this knowledge versus without it?
- How should this knowledge be structured so a reasoning engine can reliably access and apply it?
- Does this knowledge reveal a capability gap in a current cognitive function?
- Should this knowledge change how a cognitive function approaches its core responsibilities?

**Output types:** Cognitive function briefing, context architecture update, LUMIAION knowledge integration protocol, function performance standard update.

**Example (RP-001 → AI):** The machine consciousness question (Q-003) has direct implications for how LUMIAION operates: LUMIAION should not assert consciousness or deny it. The Foundation's Class Q institutional position must be embedded in LUMIAION's operating standards for any question touching AI sentience.

**Example (RP-002 → AI):** The optimal external memory (vault) architecture for LUMIAION is informed by the hippocampal index theory — the vault should function as an addressable, cue-triggered index of distributed representations, not as a flat file system. This has architectural implications for how LUMIAION's context briefings are structured.

---

### Dimension 7 — Founder Implications

**Question:** How should this knowledge influence the Founder's strategic thinking and decision-making?

**Scope:** Strategic frameworks the Founder uses, decision heuristics that should be revised, mental models that should be updated, new strategic options that this knowledge reveals, Founder phenomenological integration (for consciousness and contemplative research).

**Guiding sub-questions:**
- What decisions does the Founder regularly make that this knowledge should inform?
- Does this knowledge reveal that a prior strategic assumption was incorrect?
- Does this knowledge open a strategic option the Foundation was not previously aware of?
- For contemplative/phenomenological domains: how does this knowledge relate to the Founder's direct practice?

**Output types:** Founder strategic brief, decision framework update, strategic implications note, Founder phenomenological integration note.

**Note:** This dimension addresses the Founder directly, not the institution in the abstract. It is the most personal of the eight dimensions and requires the most care to be genuinely useful rather than generic.

---

### Dimension 8 — Future Research

**Question:** What new Research Programs naturally emerge from this knowledge?

**Scope:** New research questions that this program opens, domains that were identified as out-of-scope and now warrant their own commission, gaps in current knowledge that require investigation, cross-domain intersections that deserve dedicated programs.

**Guiding sub-questions:**
- What questions does this program answer in ways that immediately raise new questions?
- What was explicitly placed out-of-scope that should now have its own program?
- What cross-domain intersections does this program illuminate that require dedicated investigation?
- What open questions (Class Q) in this program are tractable enough to warrant a Research Commission?

**Output types:** Research Commission (RC-series) draft, Research Debt Register update, new RP-series placeholder.

**Note:** This dimension connects the end of one intelligence lifecycle back to the beginning of the next. It is how the Foundation's research portfolio compounds.

## Translation Activation Standards

Not all eight dimensions are active for every program. The following standards govern dimension activation:

| Dimension | Always Active? | Activation Condition | Deferred If |
|-----------|---------------|---------------------|-------------|
| 1 Institutional | No | Canonical Synthesis reveals gap or opportunity in Foundation structure | No structural implication identified after explicit evaluation |
| 2 Engineering | No | Knowledge defines requirements for a buildable system | No system can be specified from current knowledge |
| 3 Knowledge Graph | **Yes** | Always — every program produces nodes and edges | — |
| 4 Educational | **Yes** | Always — every program requires curriculum design | — |
| 5 Product | No | Knowledge enables something valuable to external users | Foundation is not yet in a product-building phase; no external opportunity identified |
| 6 AI | **Yes** | Always — cognitive functions must inherit research knowledge | — |
| 7 Founder | **Yes** | Always — the Founder's strategic thinking must evolve with the Foundation's knowledge | — |
| 8 Future Research | **Yes** | Always — every program opens new questions | — |

**Minimum always-active dimensions:** 3 (Knowledge Graph), 4 (Educational), 6 (AI), 7 (Founder), 8 (Future Research).

**Conditionally active dimensions:** 1 (Institutional), 2 (Engineering), 5 (Product).

A Translation that activates fewer than five dimensions without explicit deferral documentation is incomplete.

## The Translation Standard

A Translation is complete when:

**For each active dimension:**
- [ ] The specific implication is stated, not merely listed
- [ ] The required output type is identified
- [ ] The output is produced or a Research Debt entry is created with timeline
- [ ] The output is connected to the knowledge graph and Vault

**For each deferred dimension:**
- [ ] The deferral is explicitly documented
- [ ] The reason for deferral is stated (not applicable / insufficient evidence / future phase)
- [ ] A review trigger is set (when should this deferral be revisited?)

**Overall:**
- [ ] Translation Review completed (see Translation Review Process)
- [ ] Translation artifacts committed to Vault
- [ ] Founder has acknowledged the Translation outputs
- [ ] LUMIAION institutional memory updated with Translation insights

## Translation Quality Principles

### 1. Specificity over Generality

A Translation output is not a summary of the research. It is a specific institutional action derived from the research. "The Foundation should be aware of consciousness science" is not a Translation output. "Standing Order: LUMIAION's responses on AI consciousness must reflect the Foundation's Class Q institutional position" is a Translation output.

### 2. Transformation, Not Reference

A Translation output transforms the institution. It is not a pointer back to the research ("see the Canonical Synthesis for details"). It is a new institutional artifact produced because the research exists: a policy, a system specification, a learning sequence, a strategic brief, a research commission.

### 3. Permanence Design

Translation outputs must be designed to survive the replacement of the reasoning engine that produced them. A Translation output that depends on LUMIAION's current parametric memory — rather than on explicit institutional documents — will be lost at the next engine transition. All Translation outputs are committed to the Vault and versioned.

### 4. Uncertainty Inheritance

If the underlying research is Class M (competing models) or Class E (emerging evidence), the Translation output must reflect that uncertainty. A speculative research finding cannot produce a definitive engineering specification. Evidence class flows from research through Canonical Knowledge into Translation.

### 5. The Compounding Obligation

Every Translation output should be designed to serve future research, not just current application. Knowledge graph edges created today accelerate research programs commissioned tomorrow. Learning sequences designed today onboard future cognitive functions faster. Engineering systems built today become the infrastructure for capabilities not yet imagined. The Foundation builds for compounding, not for the immediate.

**Deviation:** No research program may be closed as complete without a Translation Record (or documented waiver approved by the Cognitive Council)
