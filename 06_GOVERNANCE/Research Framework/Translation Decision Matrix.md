# Translation Decision Matrix

*This matrix governs how each Canonical Research Program determines which of the eight translation dimensions to activate. The matrix is completed during the Pre-Translation phase, before any dimension is executed. All eight dimensions must be explicitly evaluated. Dimensions 3, 4, 6, 7, and 8 are always active. Dimensions 1, 2, and 5 require activation decisions.*

---

## How to Use This Matrix

1. Complete the **Program Profile** section
2. Work through each **Conditional Dimension** evaluation (1, 2, 5)
3. Confirm always-active dimensions (3, 4, 6, 7, 8)
4. Enter results in the **Activation Summary Table**
5. Copy the Activation Summary to the Translation Template (Dimension Activation Decision section)

## Dimension 2 — Engineering Implications

### Activation Question

> Does this research define sufficient requirements to produce an Engineering Proposal, Architecture Decision Record, or system specification for a system that the Foundation should build?

### Evaluation Checklist

| Question | Answer | Evidence Reference |
|----------|--------|-------------------|
| Does this knowledge identify a process that could be automated? | | |
| Does this knowledge reveal that an existing system is architecturally misaligned? | | |
| Does this knowledge suggest a new tool the Foundation needs to build? | | |
| Does this knowledge define the requirements for a system that does not yet exist? | | |
| Is the evidence class sufficient to specify engineering requirements? (C or M required; E possible; Q or S insufficient) | | |

### Evidence Class Gate

Engineering specifications require minimum Class E evidence. Apply this gate before activating:

| Evidence Class | Engineering Activation Status |
|---------------|------------------------------|
| C — Consensus | Full activation — specifications may be definitive |
| M — Competing Models | Conditional activation — specifications must reflect uncertainty; competing architectures should be noted |
| E — Emerging | Limited activation — preliminary specifications only; flag as subject to revision |
| Q — Open Question | **Cannot activate** — insufficient evidence to specify |
| S — Speculative | **Cannot activate** — insufficient evidence to specify |
| P — Phenomenological | **Cannot activate** — phenomenological evidence does not ground engineering requirements |

## Dimension 5 — Product Implications

### Activation Question

> Does this research enable the Foundation to build something of distinct value for external users that would not be available without this specific research approach?

### Evaluation Checklist

| Question | Answer | Evidence Reference |
|----------|--------|-------------------|
| Is there a persistent problem in the world that this knowledge addresses better than existing solutions? | | |
| Does the Foundation's unique multi-disciplinary, multi-function approach produce insights unavailable elsewhere? | | |
| Is there an external organization, community, or individual who would benefit from this knowledge in practical form? | | |
| Is the Foundation currently in a phase where product development is appropriate? | | |
| Is the evidence base strong enough to build a product claim on? (Minimum Class M required) | | |

### Foundation Phase Gate

Product development requires institutional maturity. Apply this gate:

| Foundation Phase | Product Activation Status |
|-----------------|--------------------------|
| Epoch I — Constitutional Formation | Deferred — Foundation not yet product-ready |
| Epoch II — Research Establishment | Limited — IP documentation only; no active product development |
| Epoch III — Engineering Build | Conditional — internal tools that become external products eligible |
| Epoch IV+ — Full Operations | Full activation available |
